- Em fluxo de push iOS, permissão concedida não prova entrega: sempre validar `didRegisterForRemoteNotificationsWithDeviceToken` e confirmar insert real em `device_tokens`.
- Quando o histórico remoto de migrations do Supabase estiver vazio, `supabase db push` pode tentar reaplicar tudo e falhar; nesses casos, aplicar a migration pontual via Management API evita mexer no histórico antigo.
- Em mudanças de modelagem como portfolio/carteira, validar primeiro o vínculo real no banco (`FK`, `backfill`, dados existentes) antes de puxar a UI; isso evita interpretar a feature só pela tela.
- Em importações ligadas a usuário/portfolio, validar o email exato no banco antes de preparar inserts; pequenas diferenças no endereço mudam `usuario_id`, `grupo_id` e possivelmente `carteira_id`.
- Neste projeto, validar mudanças de UI sempre em `next build` + `next start`; o `next dev` já serviu CSS quebrado/cache inconsistente e induziu falso positivo de layout.
- Ao endurecer rotas com `service_role`, primeiro trocar credenciais para `getServiceRoleKey()` e só depois fechar autenticação/autorização; isso reduz quebra de build enquanto a proteção é aplicada em etapas.
- No ambiente local deste projeto, PWA/service worker pode mascarar correções de layout e redirect; para validar UI no `localhost`, a raiz deve redirecionar no servidor e o build local não deve servir `sw.js` residual.
- Neste projeto, nunca importar `@/lib/coingecko` em componente client só para tipos; esse arquivo usa `fs/path/process.cwd()` e quebra hidratação. Tipos compartilhados devem ficar em `src/types/crypto.ts`.
- Nas páginas do dashboard, não depender de `useAuth().user` para buscar dados já protegidos por API/middleware; se a sessão do browser demorar a hidratar, a UI fica vazia mesmo com o backend respondendo corretamente.
- Em rotas protegidas deste projeto, não deixe páginas client nascerem zeradas e dependerem só de `useEffect`; para estabilidade local, entregue `Home`, `Crypto` e `Carteira` já com dados do servidor e use o client apenas para refresh/interação.
- Quando uma tela protegida já nasce com dados SSR, nunca faça refetch automático no primeiro useEffect; se a sessão do browser ainda não estabilizou, o fetch segue redirect para /signin e sobrescreve a UI com estado vazio.
- Para páginas protegidas com seleção de contexto, use query string como fonte principal (`carteira_id`) e deixe `localStorage` apenas como conveniência; isso reduz ambiguidade e evita drift entre telas.
- Em telas do App Router que recebem `initialData` de Server Component, sincronize o estado client quando as props mudarem; só `useState(initialData)` na montagem não acompanha troca de `searchParams`/URL.
- Em fluxos com contexto selecionado (ex.: carteira), nunca chame refresh genérico sem repassar o `carteira_id`; tarefas automáticas como geração de snapshots podem silenciosamente voltar para a carteira padrão do usuário.
- No app iOS, a seleção de portfolio precisa ser persistida em um store simples compartilhado e reaplicada em cada tela relevante; se `Crypto`, `Carteira` e `Home` não lerem a mesma fonte, o usuário sente que cada aba está olhando para uma carteira diferente.
- No app iOS, ao permitir trocar portfolio dentro de um modal de administração, sincronize imediatamente os campos do formulário com o portfolio recém-selecionado; sem isso, o usuário pode editar achando que está em um portfolio e salvar em outro.
- Em alertas recorrentes por cruzamento, `cooldown` não pode simplesmente pular o ciclo; mesmo durante o resfriamento, a function precisa continuar atualizando `last_price`, senão o próximo cruzamento é calculado com preço velho e dispara errado.
- Quando o histórico remoto de migrations do Supabase está vazio, aplique correções pontuais em um workdir temporário contendo só a migration alvo; isso evita que `db push` tente reaplicar todo o projeto remoto.
